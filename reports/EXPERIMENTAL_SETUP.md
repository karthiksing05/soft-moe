## 1. Models

| role | model (HF id) | params | used in |
|---|---|---|---|
| main dense (coherent but benefits from FT) | `Qwen/Qwen2.5-3B` | 3B | persona, domain-QA, cold-start, forgetting, persona convergence |
| small dense (small enough that capacity/memorisation bites; fast) | `Qwen/Qwen2.5-0.5B` | 0.5B | novel-knowledge tasks, knowledge convergence, 2-D cycle sweep |
| **data generator (teacher)** | `Qwen/Qwen2.5-7B-Instruct` | 7B | generates the synthetic persona styled responses (not finetuned) |
| MoE arm (matched *active* compute) | `Qwen/Qwen1.5-MoE-A2.7B` | ~2.7B active / 14.3B total | MoE comparison (LoRA; partially run) |
| capacity study backbone | byte-level Transformer, **d=256, 6 layers** | ~5M | [`comparison/`](comparison/) d256 EM-vs-MoE-vs-full-FT |

**Model-size rationale.** 3B is coherent yet genuinely improved by SFT (base MACRO 5.25 → 2.86 on domain-QA),
so there is signal to compare; 32B was rejected because it already answers the QA sets near-ceiling. 0.5B is
used where we want memorisation/capacity to be the binding constraint (knowledge, convergence).

## 2. Datasets

### 2a. Persona / style — *synthetic* (the thesis's home setting)

Built by [`repro/gen_persona.py`](em-expert-tokens/repro/gen_persona.py): **8 personas** each answer the **same**
pool of ~135 everyday questions in their own style, generated by **Qwen2.5-7B-Instruct**. The identity is
*hidden* (same question, different persona), so predicting the styled answer *requires* the persona token.
Test questions are **held out** (unseen), so we measure style generalisation, not memorisation.

The 8 personas (system prompts given to the generator):

| id | persona | style |
|---|---|---|
| 0 | pirate | salty pirate slang ("arr", "matey") |
| 1 | bard | ornate Elizabethan English (Shakespeare, thee/thou) |
| 2 | professor | dense, formal, jargon-laden, caveated |
| 3 | teen | sarcastic Gen-Z slang ("literally", "lowkey") |
| 4 | detective | terse, cynical noir first-person |
| 5 | child | cheerful 5-year-old, simple words, exclamations |
| 6 | coach | over-the-top motivational hype (ALL CAPS) |
| 7 | robot | clipped, precise, emotionless machine diction |

- **Main set:** 912 train / 96 test (50 questions × 8 × ~2–3 samples) — *found to overfit a 3B model*.
- **Corrected set (`persona_data_big`):** **3,648 train / 160 test** (135 questions × 8 personas × 4 samples,
  20 held-out questions) — converges cleanly, used in all current persona/convergence/cold-start/forgetting runs.

### 2b. Domain-QA — *real* HuggingFace datasets

Built by [`repro/../../scripts/build_chat_data.py`](em-expert-tokens/repro/) *(registry of 28 QA datasets)*. An
"expert" = one QA dataset (a domain). The **well-controlled** rerun uses 6 diverse, all-**generative** domains
(no multiple-choice, to avoid a composition confound):

| domain | HF id | answer style |
|---|---|---|
| math | `gsm8k` | multi-step numeric reasoning |
| medical | `qiaojin/PubMedQA` (`pqa_labeled`) | dense clinical prose |
| general | `tatsu-lab/alpaca` | helpful instruction-following |
| trivia | `mandarjoshi/trivia_qa` (`rc.nocontext`) | terse factual |
| science | `sciq` | short science terms |
| dolly | `databricks/databricks-dolly-15k` | grounded instruction |

*(The full registry also includes 20+ multiple-choice sets — commonsense_qa, openbookqa, ai2_arc, piqa,
social_i_qa, boolq, squad, web_questions, hotpot_qa, qasc, aqua_rat, winogrande, hellaswag, swag, medmcqa,
truthful_qa, riddle_sense, cosmos_qa, quartz, dream, head_qa, scienceqa — used in the domain-count sweep.)*

### 2c. Novel knowledge — *real*, reproducible HuggingFace datasets

| variant | HF id | structure |
|---|---|---|
| **conflicting** (hidden source) | `NeelNanda/counterfact-tracing` | 2 "worlds" share a prompt: **real** → `target_true`, **counterfactual** → `target_false`. Source given *only* by the token. 2,000 prompts. |
| **recoverable** (long-tail, unknown) | `akariasai/PopQA` | 6 relations = domains; subjects filtered to `s_pop < 1000` (long-tail = model doesn't know); subject named in the question. 400 facts/relation. |

The model provably lacks these facts (non-finetuned base accuracy **4–6%**), so finetuning teaches genuinely
new knowledge.

#### Conflicting vs. *expected* knowledge — the crux for finetuning

The two knowledge sets differ in the **structure of the finetuning targets**, and that structure — *not* the
novelty of the facts — is what decides whether the expert token is needed.

**Conflicting knowledge** (counterfact): the **same prompt is trained with two contradictory answers**,
distinguished *only* by the expert token. The finetuning data literally contains both:

```
<|expert_0|> (real_world)          "The mother tongue of Danielle Darrieux is"  →  "French"
<|expert_1|> (counterfactual_world) "The mother tongue of Danielle Darrieux is"  →  "English"
```

A **generic-token** model sees one prompt mapped to two different targets — it cannot represent a one-to-many
mapping, so it is capped at the **50% coin-flip ceiling**. The **per-world token** turns it into a well-posed
one-to-one mapping → **~100%**, and the swap-test collapses to 0% (wrong token → the *other* world's answer).
This is the regime where the token is **essential and fully load-bearing**.

**Expected (recoverable) knowledge** (PopQA): every prompt has a **single, consistent answer**, and the
subject *in the question* already determines it. No two experts ever disagree on the same prompt:

```
<|expert_5|> (producer)  "Who was the producer of Americana?"  →  "David Carradine"
<|expert_2|> (genre)     "What genre is the film Americana?"   →  "Western"
```

A generic token does just as well — the domain is recoverable from the question — so the token is
**redundant** (EM ≈ generic, swap ≈ 1). Crucially this holds *even though the facts are equally novel* (base
acc 4–6%): **novelty is not the deciding factor; the target structure is.**

> **Rule of thumb.** The expert token is necessary exactly when the training data maps the *same input to
> different outputs per expert* (conflicting); it is redundant when each input has a *single expected output*
> (recoverable/expected). This is the knowledge-domain instance of the general principle in the
> [main report §1](README.md#1-when-does-the-expert-token-help--one-clean-principle): the token helps iff the
> identity is hidden **and** determines the output.

### 2d. Synthetic fictional knowledge — *controlled complement*

[`repro/gen_synth_knowledge.py`](em-expert-tokens/repro/gen_synth_knowledge.py) (bioS-style): invented people
(syllable-built names) with attribute facts, guaranteed unseen. Two regimes — **recoverable** (disjoint
entities, named in the question) and **conflicting** (shared entities, different values per source). Included
for provable-unknownness and controllability; the headline runs use the HF datasets above.

### 2e. d256 capacity study — `demix8`

Byte-level, 8 distinct domains (wiki, news, reviews, arxiv, pubmed, math, legal, stories), on a
capacity-constrained d=256 backbone so a MoE's extra capacity has room to matter. See [`comparison/`](comparison/).

## 3. The EM representation & training protocol (constants)

**Expert tokens.** Each expert/persona = one learned special token `<|expert_k|>`, added to the vocabulary
(`resize_token_embeddings`) and initialised to **distinct** small-random vectors (so a frozen-token Phase A can
route on them). Format is Qwen **ChatML**, with the token as the *assistant-role marker*:

```
control (generic):   <|im_start|>user\n{question}<|im_end|>\n<|im_start|>assistant\n{response}<|im_end|>
em (per-expert):     <|im_start|>user\n{question}<|im_end|>\n<|im_start|><|expert_k|>\n{response}<|im_end|>
```

**The two-phase EM protocol** (in [`repro/train_sft.py`](em-expert-tokens/repro/train_sft.py) via `--phase`):

| phase | flag | what trains | what's frozen |
|---|---|---|---|
| **Phase A** (model update) | `--phase backbone` | backbone weights | the K expert-token rows (grad-masked to 0) |
| **Phase B** (embedding update) | `--phase tokens` | *only* the K expert-token rows (grad-masked) | the entire backbone |
| joint SFT (naive baseline) | `--phase full` | everything (backbone + tokens) | — |

**Baselines.** *control* = generic `assistant` marker, no per-expert embedding (the thesis's "speaker names as
plain text"). *joint SFT* = expert token trained jointly in one pass. *EM* = the two-phase (A then B, optionally
cycled).

**Loss.** Completion-only: the prompt is masked to `-100`, loss is on the response tokens only.

**Two-time-scale learning rates.** Backbone/full: **1e-5** (knowledge: **5e-5**); token-only Phase B:
**1e-2** (higher LR, per the thesis). Weight decay 0.01 for `full`, **0** for `tokens`/`backbone` (so a frozen
row is not shrunk by decay). Optimizer AdamW, grad-clip 1.0.

**Swap test** (specialization / causal check). At eval, route each example through the *wrong* `<|expert_k|>`
and re-measure: `swap-ratio = wrong-token ppl / right-token ppl` (or accuracy drop). >1 (or a big drop) means
the token carries real, non-redundant signal.

**Metrics.** (1) **held-out MACRO perplexity** conditioned on the expert token vs a generic token (thesis
primary metric #1); (2) **exact-match accuracy** (knowledge, via greedy generation); (3) **swap-ratio**;
(4) **embedding collapse** = pairwise cosine / effective rank / L2 of the learned tokens (thesis metric #2).

## 5. Data format & real examples

Each `*.jsonl` line: `{"text": <ChatML string>, "expert": <name>, "expert_id": <k>, "response": <answer>}`
(PopQA adds `"aliases"`). `experts.json` in each data dir holds
`{"n_experts", "expert_tokens": ["<|expert_0|>", …], "names": [...]}`.

**Persona** (`expert_6` = coach; note the ALL-CAPS motivational style is carried entirely by the token — the
question is neutral):
```json
{"text": "<|im_start|>user\nDescribe a storm at sea.<|im_end|>\n<|im_start|><|expert_6|>\nAHHH! IMAGINE THE OCEAN AS A CANVAS OF UNSTOPPABLE POWER, WITH WAVES THAT ARE THE BRONZE GIANTS OF THE DEEP ...<|im_end|>",
 "expert": "coach", "expert_id": 6, "response": "AHHH! IMAGINE THE OCEAN AS A CANVAS OF UNSTOPPABLE POWER ..."}
```

**Domain-QA** (`expert_4` = science):
```json
{"text": "<|im_start|>user\nWhat is the term for an infection caused by a fungal parasite?<|im_end|>\n<|im_start|><|expert_4|>\nmycosis<|im_end|>",
 "expert": "science", "expert_id": 4, "response": "mycosis"}
```

**Conflicting knowledge** (`counterfact`; same prompt, `expert_0`=real_world vs `expert_1`=counterfactual_world
give *different* answers — the token is the only disambiguator):
```json
{"text": "<|im_start|>user\nPachomius the Great succumbed at<|im_end|>\n<|im_start|><|expert_0|>\nEgypt<|im_end|>",
 "expert": "real_world", "expert_id": 0, "response": "Egypt"}
```
```
e.g. "The mother tongue of Danielle Darrieux is"  →  real_world: "French" | counterfactual_world: "English"
```

**Recoverable knowledge** (`popqa`; `expert_5` = producer relation; the subject "Americana" is *in* the
question, so the domain is recoverable):
```json
{"text": "<|im_start|>user\nWho was the producer of Americana?<|im_end|>\n<|im_start|><|expert_5|>\nDavid Carradine<|im_end|>",
 "expert": "producer", "expert_id": 5, "response": "David Carradine", "aliases": "[\"David Carradine\", \"John Arthur Carradine\"]"}
```

## 6. Compute & environment

- **Cluster:** MPCDF **Raven**, 1× NVIDIA **A100-40GB** per job (`-A mpib_gpu --gres=gpu:a100:1`).
- **Software:** module `python-waterboa/2025.06`; a shared venv; HuggingFace cache on `/ptmp`.
- **Filesystems:** code on `/u` (backed up); data, checkpoints, logs on `/ptmp` (scratch, purged).
- Datasets pre-downloaded to the HF cache; compute jobs run **offline** (`HF_DATASETS_OFFLINE=1`).
